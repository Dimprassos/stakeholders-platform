🚀 Implementation Plan: Sponsorship Management Platform
 
1. The Technology Stack
 
  - Frontend: React.js (bootstrapped with Vite), React Router v6 (for SPA

    navigation), Tailwind CSS (for styling), React Hook Form (for form

    handling).

  - Backend: Node.js with Express.js.

  - Database: SQLite (.db file) managed locally using DB Browser for SQLite. No

    ORMs—just raw SQL queries (sqlite and sqlite3 npm packages).

  - Authentication: JWT (JSON Web Tokens) for Admin login. Node's built-in

    crypto module for generating Sponsor magic links.

  - Email Emulation: Nodemailer configured with Ethereal (captures emails

    locally and provides a web link to preview them without spamming real

    inboxes).

  - File Uploads: multer (Node.js middleware) to save uploaded sponsor logos

    locally, storing the file path in SQLite.
 
2. Database Schema (SQLite)
 
You will create a sponsorship.db file. Here are the core tables and raw SQL

structures you will execute:
 
1.  admins: id (PK), email (TEXT), password_hash (TEXT).

2.  packages: id (PK), name (TEXT), description (TEXT), price (REAL), benefits

    (TEXT - stored as JSON string), is_active (BOOLEAN).

3.  candidates: id (PK), company_name (TEXT), contact_email (TEXT), status

    (TEXT: PENDING, INVITE_SENT, ACCEPTED, DETAILS_SUBMITTED), package_id

    (INTEGER, FK), magic_token (TEXT), token_expires_at (DATETIME).

4.  sponsor_details: id (PK), candidate_id (INTEGER, FK), legal_name (TEXT),

    billing_address (TEXT), vat_number (TEXT), website_url (TEXT),

    logo_file_path (TEXT).
 
3. Backend API Architecture (Node.js & Express)
 
The Express server will run on localhost:5000 and act as the bridge between your

SQLite DB and the React frontend.
 
A. Admin Routes (Protected by JWT Middleware)
 
  - POST /api/auth/login: Verifies admin email/password against the admins

    table. Returns a JWT.

  - GET /api/admin/packages: Runs SELECT * FROM packages to display in the CMS.

  - POST /api/admin/packages: Runs INSERT INTO packages... to create a new tier

    (e.g., Gold, Silver).

  - GET /api/admin/candidates: Runs SELECT * FROM candidates to populate the

    sponsor pipeline table.

  - POST /api/admin/candidates/:id/invite:

      - Generates a 32-char crypto token.

      - Runs UPDATE candidates SET magic_token = ? ...

      - Uses Nodemailer + Ethereal to send the email containing the magic link

        (http://localhost:5173/invite/{token}).
 
B. Sponsor Routes (Public, Protected by Magic Token)
 
  - GET /api/sponsor/invite/:token:

      - Runs SELECT * FROM candidates WHERE magic_token = ?.

      - Validates expiration date.

      - Returns candidate and assigned package details.

  - POST /api/sponsor/invite/:token/accept: Updates candidate status to

    ACCEPTED.

  - POST /api/sponsor/invite/:token/submit-details:

      - Uses multer to handle the logo image upload.

      - Runs INSERT INTO sponsor_details ... with billing info, VAT, and the

        logo file path.

      - Updates candidate status to DETAILS_SUBMITTED.
 
4. Frontend Architecture (React.js + Tailwind)
 
The React application will run on localhost:5173 (Vite's default port). It is

split into two distinct areas via React Router.
 
A. Admin CMS (The Control Panel)
 
  - /admin/login: Simple centered card with Email/Password. Stores JWT in

    localStorage on success.

  - /admin/dashboard: A layout with a sidebar (Navigation) and main content

    area. Shows top-level metrics (e.g., Total Invites Sent, Total Accepted).

  - /admin/packages: A page to view/add/edit sponsorship packages.

  - /admin/candidates: The CRM Table. Shows columns: Company, Email, Assigned

    Package, Status.

      - Action: A "Send Invite" button next to each candidate that triggers the

        Ethereal email.
 
B. Sponsor UI (The Client Experience)
 
  - /invite/:token: The Proposal Landing Page. Uses useEffect to fetch package

    data based on the token. Displays a welcome message ("Welcome, [Company

    Name]") and Package perks. Has a big "Accept Package" button.

  - /invite/:token/form: The Onboarding Wizard. Displays form fields for Legal

    Name, Billing Address, VAT Number, and a file input for their Logo.

  - /invite/success: A final "Thank You" screen informing them that the admin

    will be in touch with the invoice.
 
5. End-to-End User Flow
 
Flow 1: Admin Prepares the Pitch
 
1.  Admin opens React app (/admin/login) and logs in.

2.  Admin goes to Packages and creates a "Diamond Sponsor - $5,000" package.

3.  Admin goes to Candidates and adds "TechCorp Inc." with the email

    ceo@techcorp.com, assigning them the Diamond package.

4.  Admin clicks "Send Invite".

5.  Backend generates the magic link and sends it. Terminal outputs the Ethereal

    link for you to view the email.
 
Flow 2: Sponsor Accepts the Pitch
 
1.  Sponsor clicks the link in their Ethereal email:
http://localhost:5173/invite/abc123xyz...

2.  React Router reads abc123xyz..., pings the Node server to validate it, and

    displays the Diamond Package details.

3.  Sponsor clicks "Accept Package".

4.  React routes to the form step. Sponsor types in their Billing Address, VAT

    number, and uploads their PNG logo.

5.  Sponsor clicks "Submit".

6.  Backend saves the image to an uploads/ folder, saves the form data to

    SQLite, and updates the status.

7.  React shows a Success Page. Admin can now open DB Browser (or their Admin

    CMS) and see TechCorp's status as DETAILS_SUBMITTED.

  - Phase 4: Sponsor UI & File Uploads (Day 5)

  - Phase 5: Polish & Test (Day 6)

1. UI/UX & Accessibility Standards (The "Cross-Cutting" Core)
 
Before building features, the frontend architecture (React + Tailwind) must

support these global conditions:
 
  - Theme Matrix: Native support for Light / Dark / System themes. In Dark mode,

    brand colors shift to "pale" variants (e.g., pale purple for Admin, pale

    gold for Sponsors) for legibility. No "light-only" flash on reload.

  - Text-Size & Zoom: A global text-size control (Smallest, Default, Largest).

    UI must intelligently stack (not cram) at Largest text size. Must

    survive 200% browser zoom without horizontal scrolling.

  - Reduced Motion: A global toggle. When ON, route transitions, modal fade-ins,

    and carousel snaps become instant.

  - Device Breakpoints:

      - P / Pℓ (Phones): Hamburger menus, single-column forms, bottom horizontal

        carousels.

      - T / Tℓ (Tablets): Table-to-Card structural shifts, 2-column forms.

      - L / D (Desktop): Full data tables, side-by-side sticky panels.
 
2. Expanded Database Schema (SQLite)
 
To support multi-event management, visibility toggles, and robust tracking.
 
  - events: id, name, description, start_date, end_date, location, is_public.

  - packages: id, event_id, name, price, benefits (JSON).

  - candidates (The CRM): id, event_id, company_name, contact_email, status

    (PENDING, INVITE_SENT, ACCEPTED, DECLINED, DETAILS_SUBMITTED), magic_token,

    token_expires_at.

  - sponsor_details: id, candidate_id, legal_name, billing_info, logo_file_path,

    website_url, is_hidden_from_public (BOOLEAN).
 
3. Backend API & Live Data Engine (Node.js + Express)
 
  - Live Updates (Server-Sent Events - SSE): Instead of refreshing, the Admin

    dashboard uses SSE. When a sponsor accepts a package or uploads their logo,

    the Express server pushes an event to the React Admin app, updating the CRM

    table and status counts live.

  - Draft Survival (Auto-save): Endpoints to save partial sponsor forms. If a

    sponsor closes the tab halfway through uploading their details, returning to

    the magic link restores their typed data.

  - Data Integrity (Safe Deletes): If an Admin deletes an event, a database

    transaction ensures all associated packages, candidates, and uploaded logo

    files (via multer) are safely wiped or archived without creating orphans.
 
4. Admin CMS: Organizer Dashboard
 
  - Multi-Event Dashboard: Admins see cards for each event showing status counts

    (e.g., "10 Accepted, 5 Pending"). Clicking "More details" expands inline to

  - Responsive CRM Roster:

      - Desktop (L/D): Shows a full data table.

      - Mobile/Tablet (P/T): Morphs into stacked cards with a dedicated "Sort &

        Filter" dropdown.

      - Live Search: Instant filtering by Name, Email, or Status ("Details

        Submitted").

  - CSV / XLSX Import & Export: Admins can bulk-import candidates from an Excel

    "imported/skipped" counts.

  - Live Email Template Editor: When drafting an invite, the screen splits (or

  - Send Invites: Bulk "Send" action that pushes to Nodemailer/Ethereal and
